## Functional overview

For each frame and each body part, the program first builds all plausible cross-camera combinations of the 2D detections. With three cameras and two DLC identities, it tests all triplets such as `(cam0 mouse0, cam1 mouse1, cam2 mouse0)` and all two-camera pairs. This is the key mechanism for correcting 2D identity confusion: the program does **not trust the DLC individual labels** to be consistent across views. Instead, it tries all identity combinations and asks which ones are geometrically plausible in 3D. ţłÇfileciteţłéturn3file0ţłü

### Stage A: 3D geometric validation

Each candidate pair or triplet is converted into camera rays using the calibrated camera models. The candidate is accepted only if it passes several 3D/multiview consistency tests:

1. **Ray-angle gate.** The camera rays must intersect at a sufficiently non-degenerate angle. Very small triangulation angles are rejected because they give unstable depth estimates.

2. **Closest-approach gate.** The rays from different cameras must pass close to one another in 3D. A mismatched pair, such as the head of mouse 0 in one camera and the head of mouse 1 in another, will often produce rays that do not meet in space.

3. **Cage/bounding-box gate.** The initial 3D point must lie within, or near, the known cage/world volume.

4. **Fast reprojection gate.** The estimated 3D point is projected back into the original camera images. If the reprojection error is too large, the hypothesis is rejected.

5. **Bundle-adjusted reprojection error.** Accepted initial hypotheses are refined by a Ceres point-only bundle adjustment that minimizes reprojection error. The final RMS pixel error must be below the configured threshold. Triplets are given a selection bonus over pairs, so the code prefers three-view evidence when it is plausible. ţłÇfileciteţłéturn3file0ţłü ţłÇfileciteţłéturn2file0ţłü

After scoring all accepted hypotheses, the program chooses up to two **disjoint** 3D hypotheses per body part, corresponding to the two mice. ÔÇťDisjointÔÇŁ means the same 2D observation cannot be used in two reconstructed 3D points. Any remaining 2D detections that cannot be triangulated are retained as single-view observations for later identity assignment. ţłÇfileciteţłéturn2file1ţłü

### Stage B: identity cleaning across mice

Stage B assigns each accepted 3D point, and later each remaining single-view point, to mouse 0 or mouse 1. It uses several cues.

The first cue is **segmentation support**. If segmentation masks are provided, each 2D detection receives local Gaussian-weighted support for background, mouse 0, and mouse 1. An observation ÔÇťvotesÔÇŁ for a mouse if the local mask weight exceeds a threshold; a triangulated item accumulates votes over all its contributing views. If the votes support only one mouse, the item gets an ÔÇťexclusiveÔÇŁ identity cue. ţłÇfileciteţłéturn2file19ţłü ţłÇfileciteţłéturn4file0ţłü

The second cue is **temporal continuity**, especially for the anchor body parts `head` and `tail_root`. The code keeps previous and previous-previous 3D positions per mouse and predicts the next position by constant-velocity extrapolation. It compares two global hypotheses: either the current anchor identities are consistent with the previous frame, or they are swapped. If the cost difference is clear, the code chooses `NOSWAP` or `SWAP`; if not, it carries the previous decision briefly or marks the state unknown. This is designed to repair frame-to-frame identity flips. ţłÇfileciteţłéturn4file0ţłü

The third cue is a **per-mouse body model** built from the assigned anchors. Once head and tail-root are assigned, the program constructs for each mouse a centroid and a headÔÇôtail axis line. Other 3D body parts are assigned using a cost that combines: segmentation exclusivity, temporal continuity for that body part, distance to the mouse centroid, and distance to the mouse body axis. Thus, a point near mouse 0ÔÇÖs predicted skeleton and body axis is favored for mouse 0 even if the raw DLC label says otherwise. ţłÇfileciteţłéturn4file0ţłü

The fourth cue is an optional **graph/skeleton plausibility model**. If a graph model is supplied, the program loads named 3D template nodes and edges in millimeters.
