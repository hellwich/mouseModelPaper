Your traceback dies at `import torch`, before the script can touch the input files. In your script, `torch` is imported at top level, so failure there happens before argument parsing or data loading. fileciteturn0file1 Your protocol also shows that this same project successfully trained and predicted in February 2026, including the `out_pair_train2` workflow, so the code path itself used to work. fileciteturn0file2

The specific symbol `iJIT_NotifyEvent` is a known PyTorch binary/runtime compatibility failure. PyTorch maintainers documented that `import torch` can fail with exactly this error when PyTorch is combined with MKL 2024.1+, and that downgrading MKL to 2024.0 resolves it. Independent reports around Intel’s PyTorch ecosystem say the same thing: newer oneAPI/MKL runtimes broke binaries built against the older runtime. citeturn1search0turn4search5turn4search8

Why it probably happened in your case: your `environment.yml` mixes `pytorch`, `nvidia`, and `conda-forge`, while leaving most package versions unpinned. That is workable initially, but later `conda install`, `mamba install`, or `pip install` operations in the same env can silently replace low-level libraries such as `mkl`, `intel-openmp`, or related runtime pieces. Once that happens, PyTorch may still be installed, but no longer loadable. fileciteturn0file0 citeturn2search0turn1search0

Assistant: This is no longer the original failure. The new environment is not broken; it is just incomplete. `mouse_sim_fix` contains only Python and pip, because the big `conda install ...` step failed, so `ModuleNotFoundError: No module named 'torch'` is exactly what one should expect there.

The real blocker is the `opencv` solve, and the reason is visible in your command structure. With **strict channel priority**, Conda ignores lower-priority channels for a package name if that package also exists in a higher-priority channel. Since you asked for `opencv` in the same transaction as `-c pytorch -c nvidia -c conda-forge --strict-channel-priority`, Conda is allowed to prefer an `opencv` from the `pytorch` channel instead of `conda-forge`, and your solver output shows it chasing an ancient `opencv-4.2.0-py36_5` / `libopencv` combination. That is a channel-mixing artifact, not a sign that PyTorch 2.2.2 itself is unavailable. citeturn0search1turn0search7turn0search13

Also, your chosen PyTorch stack is valid on paper. PyTorch’s official previous-versions page lists exactly this Conda combination for Linux/Windows: `pytorch==2.2.2 torchvision==0.17.2 torchaudio==2.2.2 pytorch-cuda=12.1 -c pytorch -c nvidia`. citeturn3view1

So the fix is: **do not install `opencv` in the same strict-priority solve as PyTorch**.

The current deep model does **not** use mouse 0’s own body/model coordinate system as the reference frame. Instead, for each time step it builds a **pair frame** from the two mice’s `tail_root` positions in world coordinates: the origin is the midpoint between the two tail roots, the x-axis points from mouse 0 tail root to mouse 1 tail root projected onto the xy-plane, and the z-axis is fixed to world up `(0,0,1)`. Then **all keypoints of both mice** are translated and rotated into that pair frame. This is done in `pair_frame_transform()` and `to_pair_frame()`. fileciteturn8file0

So the preprocessing removes a lot of global information. Absolute cage position is removed because the origin is recentered to the midpoint each frame. Absolute heading in the cage is also removed because the pair is rotated so that the inter-tail-root direction becomes the x-axis each frame. What remains is the **relative configuration of the two mice**, including each mouse’s pose relative to the pair frame, the distance between them, and changes of that relative configuration over time. That is exactly why the representation is well suited for the interaction-vs-noninteraction decision. fileciteturn8file0

The actual feature vector is built in `make_pair_features()`. It flattens all transformed coordinates for both mice, so the static position part has size `2*K*3`. If `use_vel=True`, it also appends first differences over time of those transformed coordinates, adding another `2*K*3`. Finally it adds two scalar extras: `nose_dist` and `tail_dist`, both measured in the pair frame. So with velocities enabled, the feature dimension is `12*K + 2`. fileciteturn8file0

A subtle point: these “velocities” are **not global world-space velocities**. They are frame-to-frame differences **after** the pair-frame transform. So they encode motion in a moving, rotating relative coordinate system. If both mice translate together across the cage while keeping the same mutual configuration, that common global translation is largely removed. If the whole pair rotates together in the cage without changing mutual arrangement, that rotation is also largely removed. What remains is motion of body parts **relative to the pair frame**, which is exactly the motion signal the model was trained on. fileciteturn8file0

Another non-obvious detail: `head` is checked for existence, but in the deep feature construction it is not actually used numerically. `make_pair_features()` uses the full flattened transformed coordinates of all nodes, plus `nose_dist` and `tail_dist`; the `head` argument is only validated. The `head` point matters in the geometric rule baseline, but not in the learned feature construction shown here. fileciteturn8file0

There is also a training-time symmetry mechanism that matters conceptually. `preprocess_run()` creates `X_base` and also `X_swap`, where mouse identities are swapped, the pair frame is recomputed, and the per-mouse labels are swapped too. The training dataset then randomly uses base or swapped windows. That means the model is trained to be relatively invariant to whether a given physical mouse is indexed as 0 or 1. At inference, only `X_base` is used. fileciteturn8file1

The pair frame is computed directly and only from the two `tail_root` positions already present in `run.pos`. If `tail_root` is wrong or missing, the frame is wrong. There is an additional important issue: the loader does **not** explicitly reject missing body-part rows inside a frame/mouse. `load_coords_3d_csv_pair()` initializes the whole `pos` array with zeros and then fills only the rows that exist in the CSV. It raises an error only if an entire frame/mouse block is missing, not if some individual nodes are absent. Therefore, if a body part is occluded and omitted from the CSV, that node’s coordinate stays at `(0,0,0)` silently. fileciteturn8file0

So in practice the current software already has a kind of “implicit missing-value handling,” but it is a bad one: **missing points become zeros, not masked values**. Downstream, the deep model receives a dense fixed-length tensor with no visibility mask channel. There are three realistic ways forward.

The least invasive option is **imputation before the current preprocessing**. Convert your estimated 3D data into the expected CSV schema, then for each mouse and frame estimate a rigid or similarity transform from the visible observed points to the canonical mouse model (or use temporal interpolation / last-known pose), and fill in the missing points before calling the existing predictor. A more robust variant is to do that imputation with temporal smoothing, because framewise fitting from a sparse visible subset can be noisy. In other words: use visible points to estimate pose, propagate pose over time, and fill missing keypoints from the posed model. That is still compatible with the existing trained network because the network only sees completed coordinates afterward.

The principled long-term option is to **extend the model and retrain**. You would add visibility/missingness channels per keypoint, possibly also confidence scores from DLC/triangulation, and train with synthetic point dropout so the network learns how to interpret missing observations. That would be the cleanest way to support real occlusion. But it is a different model from the one you have now.
